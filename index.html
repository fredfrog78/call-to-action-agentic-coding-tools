The trick

Use the tool for mechanical work and fast iteration, but keep the engineering loop sacred: small diffs, colcon tests, bag replay or sim runs, and code review before anything touches a real robot.

The phase change: from autocomplete to agentic loops

“Autocomplete” helps you type. “Chat” helps you think. “Agentic” helps you do work end‑to‑end by using tools (repo search, edits, builds, tests) inside your robotics workspace.

Autocomplete

  • Speeds up local code writing
  • Low risk, low leverage
  • Still you doing every step

Chat assistant

  • Explains concepts & code
  • Great for tf/QoS/launch “why is this broken?” moments
  • Still leaves you doing the glue work

Agentic tool

  • Reads your workspace + launch/params
  • Creates patches as diffs
  • Runs colcon, tests, and iterates

The most common failure mode I hit wasn’t “AI is useless.” It was “AI is almost right.” In robotics, “almost right” becomes flaky bringup, subtle frame issues, or a regression that only shows up on hardware. The solution is not vibes; it’s a workflow: permissions, diffs, tests, bag replay/sim, and review.

What agentic tools can do in real ROS 2 autonomy stacks

The point isn’t replacing engineering judgment. It’s compressing the time between “I understand the bug” and “the fix is merged with tests and a reproducible validation.”

ROS 2 spelunking that scales

Ask for a map of packages, launch entrypoints, parameters, and runtime wiring. Great when you inherit a stack with “just run bringup.launch.py” vibes.

  • Find who publishes /cmd_vel and where it’s remapped
  • Trace perception: driver → preprocessing → detector → tracker
  • Summarize tf frames, QoS expectations, and lifecycle transitions

Launch + parameter refactors safely

Refactors across launch files, parameter YAMLs, and message schemas are painful but mechanical—perfect for an agent paired with build/test gates.

  • Normalize params.yaml structure across robots
  • Fix QoS mismatches (reliability/history/depth)
  • Keep bringup reproducible across sim and hardware

Bag replay + sim smoke tests

Turn field logs into regression tests. Let the agent scaffold, then you define “correct.”

  • Create a rosbag2-based repro for “works on robot A” bugs
  • Add headless sim smoke tests for launch and lifecycle
  • Generate edge cases (timestamps, missing tf, sensor dropouts)

“Explain the diff” for safety review

Ask it to justify each change, list hazards, and propose rollback/feature flags.

  • Hazard notes + mitigations you can audit
  • Runtime guards + parameter flags (ship “off” first)
  • Rollout checklist: sim → bag replay → staged robot test
Pick a tool your org approves

Codex, Claude, and Gemini can all be used in agentic workflows. Exact features vary by product, but the adoption pattern is the same: start read‑only, earn trust, then expand permissions (especially before you run anything near hardware).

Worked examples (real robotics repos)

Two end‑to‑end agentic workflows pulled from real CSIRO robotics code: (1) fixing a silent config bug in OHM, and (2) adding a “smart filter” to RayCloudTools to keep high‑confidence points and drop noise.

Example 1 — OHM: silent boolean parsing bug

Derived from csiro-robotics/ohm. Documented boolean strings like false/no/off are interpreted as true—a classic “config says off, behavior is on” failure.

1

Map the codebase (read‑only)

Ask the agent to find the boolean parsing logic, match it against the docs, and give you file/line references you can independently verify.

Prompt 
Read this repo in read-only mode: https://github.com/csiro-robotics/ohm

Task: verify that MapValue boolean parsing matches its documentation (true/false/yes/no/on/off).

Output:
1) the exact code path (file + line numbers),
2) a minimal failing example (a few lines of C++),
3) impact analysis: what can be silently misconfigured,
4) the smallest safe fix (no refactor).
2

Force a plan + tests

Don’t accept a patch without a plan. Ask for edge cases, rollback, and explicit tests.

Prompt 
Propose a plan to fix a critical boolean parsing bug in OHM:
- MapValue should parse "false"/"no"/"off" as false (it currently doesn’t).

Constraints:
- keep behavior identical for numeric strings ("0"/"1") and unknown strings (fallback to float != 0),
- patch should be minimal and reviewable,
- add a gtest that fails before the fix and passes after,
- include exact commands to run the specific test.
Prompt 
Before writing code: produce a concrete plan with exact file paths + line references, then propose:
- the minimal code change,
- a unit test (gtest),
- and a quick way to validate (ctest or gtest filter).

Then ask me 3 questions if anything is ambiguous.
Prompt 
Plan a minimal fix for a boolean string parsing bug in a C++ robotics library.
Include:
- exact files to change,
- acceptance criteria,
- tests to add,
- how to validate locally (cmake + ctest; no GPU dependencies).
3

Apply the patch as a diff

Keep it boring: fix the bug, add a test, and explain the diff. No refactors.

4

Run tests and validate

Make the tool run the same checks you require in CI. Treat failures as part of the loop, not a surprise.

Commands 
# from the OHM repo root (CPU-only build)
cmake -S . -B build -DOHM_FEATURE_TEST=ON -DOHM_FEATURE_OPENCL=OFF -DOHM_FEATURE_CUDA=OFF
cmake --build build -j

# run tests (or filter to your new test)
ctest --test-dir build -R ohmtest --output-on-failure
./build/bin/ohmtest --gtest_filter=MapValueBoolFromString*
Example 2 — RayCloudTools: add a “smart filter” for noisy points

Derived from csiro-robotics/raycloudtools. Ray clouds carry per‑point sensor origins (stored in the PLY normal field) and optional intensity (stored in alpha). We’ll add a filter that scores confidence and keeps the good geometry.

1

Map the pipeline (read‑only)

Find where ray clouds live, how existing denoising works, and where a new CLI tool fits into the build.

Prompt 
Read this repo in read-only mode: https://github.com/csiro-robotics/raycloudtools

Goal: add a new CLI tool (or mode) that acts as a “smart filter” for ray clouds:
- remove noisy bounded points,
- keep high-confidence bounded points,
- keep unbounded rays untouched.

Find and cite:
1) where intensity/boundedness is represented (alpha channel) and the raycloud format,
2) existing noise removal code (e.g., raydenoise) and any reusable helpers (kNN, surfels),
3) how a new tool is wired into CMake (raycloudtools/* layout),
4) how tests are structured (tests/raytest).

Output: a short architecture map + the smallest set of files to change.
2

Force a plan + tests

Make it specify the confidence score, CLI, performance bounds, and tests.

Prompt 
Propose a plan to add a “smart filter” to csiro-robotics/raycloudtools.

Implementation idea:
- compute a per-point confidence for bounded rays from:
  (a) local neighbourhood consistency (kNN distances / surfel Mahalanobis distance via Cloud::getSurfels),
  (b) intensity (alpha), when present,
  (c) boundedness (alpha==0 → always keep; it represents free space).
- keep points with confidence >= threshold (or keep-top-percent).

Constraints:
- no new dependencies (use existing Eigen + libnabo),
- deterministic output (no RNG),
- O(n log n) or better; don’t allocate huge extra memory.

Deliverables:
- new tool (suggest name: rayfilter) under raycloudtools/,
- unit test added under tests/raytest that fails before and passes after,
- exact build + test commands (cmake + ctest).
Prompt 
Before writing code: propose a concrete plan with file paths, a confidence scoring formula, and a test strategy.
Include edge cases (unbounded rays, missing intensity, tiny clouds) and performance constraints.

Then ask me 3 questions if anything is ambiguous.
Prompt 
Plan an implementation for a new raycloudtools “smart filter” CLI.
Include:
- exact files/directories to add or modify (CMake + C++),
- CLI usage examples,
- acceptance criteria (“noise removed, high confidence kept”),
- tests to add (gtest) and how to run them locally.
3

Apply the patch as a diff

Add the new tool, wire it into CMake, and keep the changes reviewable.

4

Run tests and validate

Run the unit tests and a quick CLI smoke test on a synthetic room ray cloud.

Commands 
# from the raycloudtools repo root
cmake -S . -B build -DRAYCLOUD_BUILD_TESTS=ON
cmake --build build -j
ctest --test-dir build -R raytest --output-on-failure

# smoke test: make a cloud, filter it, inspect stats
./build/bin/raycreate room 1
./build/bin/rayfilter room.ply 0.75 confidence

Start a 30‑minute pilot (low risk, high signal)

If you’re skeptical, good. This pilot is designed to produce a diff and a measurable outcome, not a debate.

Pick the right first task

  • Small surface area (1–3 files/packages)
  • Existing tests, or easy to add (ament/pytest)
  • Clear success criteria
  • Not safety-critical on day one (don’t start in the control loop)

Run the workflow (permissions first)

  1. Read-only: map packages, launch/params, and runtime wiring.
  2. Plan: steps + tests + risks + rollback (sim/bag replay if possible).
  3. Patch: small diff, explain changes.
  4. Verify: run colcon test + a quick launch/sim smoke test.
  5. Review: treat it like any PR.
Definition of done

You should end with: a reviewed diff, passing colcon tests, a minimal sim/bag validation, and a short note: time spent, surprises, and what you’d change about the workflow.

FAQ: the concerns are real — here’s how to handle them

“It hallucinates. I can’t trust it.”

Treat outputs as proposals, not truth. Use a workflow that makes mistakes cheap: require file paths, prefer diffs, and insist on tests.

  • Ask it to cite exact packages/symbols and the launch/params it relied on.
  • Ask for edge cases: missing tf, bad timestamps, QoS incompatibilities, sensor dropouts.
  • Prefer incremental diffs with fast gates: colcon test, bag replay, sim smoke test.
“What about security and secrets?”

Adopt with least privilege. Start in read‑only mode, avoid pasting secrets, and use org-approved configurations.

  • Never paste fleet credentials, OTA keys, or customer site data.
  • Be careful with maps, calibration, and logs (they can be sensitive).
  • Keep the same SAST/SCA gates you already trust.
“Will it make the stack worse over time?”

It can—if you skip review. The fix is standards and enforcement: lint, tests, code owners, and smaller PRs. Make the agent follow your conventions, and reject diffs that don’t.

  • Require the agent to explain invariants and tradeoffs.
  • Enforce formatting and static analysis (ament_lint, clang-tidy, etc.).
  • Prefer characterization tests before refactors (bag replay is your friend).
“Will I stop understanding my own system?”

Use the agent to improve understanding: ask for diagrams (ROS graph, tf tree), invariants, and “why this is safe.” If you can’t explain the diff, don’t merge it.

“Is this safe for robotics? I don’t want surprises on hardware.”

Treat agentic tools as an acceleration of your existing engineering process, not a new runtime dependency. Don’t run unreviewed changes on a real robot. Gate with sim, bag replay, tests, and staged deployments.

“Which tool should I use: Codex, Claude, or Gemini?”

Use the one your org can approve and support. For a first pilot, tool choice matters less than workflow: read-only → plan → diff → tests → review.
